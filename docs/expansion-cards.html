Hardware Expansion Cards

A detailed guide to integrating, testing, and managing various envisioned hardware add-on modules for LoRaLink.

1. General I/O Expansion (MCP23017 Card)

The MCP23017 card expands available GPIO significantly, driving up to 16 additional pins via the shared I2C bus.

Stub Implementation

#include <Adafruit_MCP23X17.h>

Adafruit_MCP23X17 mcp;

void initMCP23017() {
  // Use Heltec's pre-configured I2C for OLED avoiding Wire.begin() conflicts.
  if (!mcp.begin_I2C(0x20)) {
    Serial.println("MCP23017 Error at I2C 0x20");
    return;
  }
  // Setup Pins 0-7 as Relay Outputs
  for (int i=0; i<8; i++) mcp.pinMode(i, OUTPUT);
}

UI Representation

In the Fleet Admin or Local Webserver, expanded pins use the prefix format PIN_MCP_{ExpanderID}_{PinNum}. The UI dynamically detects the hw_registry capability bits. If the Expander bit is active, a unique "Expander Module" widget is displayed tracking all 16 IO points concurrently.

Communication Testing

To verify the MCP23017 card is communicating properly:

  1. I2C Scan: Connect a Serial Monitor and reboot the device. Ensure I2C Device found at address 0x20 appears in the boot log.
  2. Pin Toggle: Open the Local Webserver and try to toggle PIN_MCP_0_0. Verify the physical pin voltage shifts from 0V to 3.3V.

2. Environmental Sensor Card (DHT22 / BME280)

This card collects local climate data, powered down automatically via PIN_VEXT_CTRL between measurements to extend battery life.

Stub Implementation

#include <DHT.h>

#define DHTPIN 15
#define DHTTYPE DHT22
DHT dht(DHTPIN, DHTTYPE);

void readEnvironmentTask() {
  digitalWrite(PIN_VEXT_CTRL, LOW); // Turn on Power Rail
  delay(100); // Allow sensor startup

  float t = dht.readTemperature();
  float h = dht.readHumidity();

  digitalWrite(PIN_VEXT_CTRL, HIGH); // Shut down Power Rail

  // Format JSON payload for Fleet Telemetry
  String payload = "{\"temp\":" + String(t) + ",\"hum\":" + String(h) + "}";
}

UI Representation

The Fleet Admin interprets the incoming JSON telemetry and renders a "Climate Card" for that specific node. Data trends over time are tracked on the GIS map.

Communication Testing

Observe the Swarm Dashboard UI. A telemetry payload should appear every 5 minutes. If temp returns NaN, physically confirm VExt (Pin 36) is dropping to 0V during the measurement phase, supplying power to the card.

3. Relay Controller Card (110V & 12V High Power)

Enables heavy payload switching. Demands strict battery thresholds to prevent hardware brownouts.

Stub Implementation

void applyRelayState(int pin, bool state) {
  // Validate System Health before 110V shift
  if (pin == PIN_RELAY_110V && getBatteryVoltage() < 3.5) {
     Serial.println("ERR: Insufficient voltage for 110V load.");
     return;
  }
  digitalWrite(pin, state ? HIGH : LOW);
  saveToNVS("lora_hw", String(pin).c_str(), state);
}

UI Representation

Relays appear inside the "Hardware Override" dashboard. A 110V switch requires a secondary safety confirm ("Are you sure?") to prevent accidental high-current draw. Failed validations highlight the toggle border in Red.

Communication Testing

Activate a 12V relay via the Fleet Administrator. Verify a Command Log entry reading "Sent Toggle Relay to Node-01" is visible. The node should acknowledge by sending a status text reply [MSG] Relay 46 is HIGH.
