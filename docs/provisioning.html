Provisioning & Network Reliability

A comprehensive guide on OEM configuration, bulk production flashing, system verification, and the mechanisms keeping devices connected.

1. System Verification (The Ping/Pong Cycle)

Once a device is configured, it's crucial to verify its hardware capabilities and telemetry routing are functioning correctly over the Swarm network.

The Heartbeat Task

To ensure a node's Task Scheduler is alive without throwing high-voltage relays, the system relies on a non-destructive Heartbeat Task. By default, the firmware maps the orange built-in LED (Pin 35) to a recurring toggle event every 2 seconds. If the node is powered and the scheduler hasn't crashed, this LED will pulse.

Ping/Pong Telemetry Validation

Using the Swarm Dashboard or a Serial Terminal, sending a `PING` command forces a complete traversal of the network stack:

  1. Command Arrival: The device receives the MQTT/Serial payload `PING`.
  2. Command Parsing: The `CommandManager` identifies the explicit keyword.
  3. Payload Response: The device constructs a binary `MessagePacket` with the payload `Pong` and injects it back through the `MQTTManager` or `LoRaManager`.
  4. Dashboard Feedback: A successful round-trip guarantees both RX and TX pipes are structurally sound.

2. Network Persistence & Reconnect Logic

Industrial IoT requires edge devices to gracefully handle network failures, brownouts, and physical interference.

The Reconnect Loop

🔄 Non-Blocking Reconnects

LoRaLink utilizes an asynchronous, non-blocking state machine in `WiFiManager` and `MQTTManager`. If a connection drops, the device does not freeze in a `while()` loop; it schedules connection attempts every 5-10 seconds, allowing local tasks (relays/sensors) to proceed uninterrupted.

NVS State Saves

To survive hard power loss, physical pin states are flushed to identical `lora_hw` keys in Non-Volatile Storage (NVS). When the device reboots, its `DataManager` accesses the namespace and automatically reapplies the last known state to the physical GPIO headers.

Watchdog Timers (WDT)

The ESP32-S3 utilizes Hardware and Software Watchdogs. If a poorly written custom task blocks the CPU for more than 3000ms, the system intentionally "panics" and restarts the node to reset the environment.

3. OEM Configuration vs. Bulk Production Flashing

For manufacturers looking to build product lines on top of LoRaLink, two distinct stages occur: OEM Scaffolding and Bulk Flashing.

Phase A: OEM Configuration (The Gold Standard)

The goal is to generate a "Gold Master" CSV or JSON configuration file representing the finished product's capabilities.

  • Connect a fresh board to the Fleet Admin PC via USB.
  • Via the Admin → Device Settings tab, rename pins to match the chassis (e.g., Pin 5 becomes "Main_Pump").
  • Toggle `hw_registry` bits if adding an OLED or secondary modules.
  • Click Export Config. You now possess an immutable `heltec_product_A.json` file.

Phase B: Bulk Production Flashing

Once you possess the "Gold Master", the objective shifts to copying it onto 100+ devices rapidly on an assembly line.

Rather than using the WebApp click-interface, operators should use the NVS Partition Image Generator to construct a binary block of NVS data containing the JSON mappings.

# Generates custom partition bin
python tools/nvs_generator.py heltec_product_A.json my_nvs.bin

# Flashes firmware AND config in one command
esptool.py --port COM7 write_flash 0x10000 firmware.bin 0x210000 my_nvs.bin
⚠️ Individual Provisioning

Even though Bulk Flashing sets all settings perfectly, it duplicates the `loralink:dev_name`. For large swarms, ensure each device is booted at least once to assign a unique suffix (e.g., `Node-A1`, `Node-A2`), or seed the UUID via the ESP32 MAC address upon first boot.
